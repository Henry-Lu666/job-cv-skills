# CDP稳定后"应抓尽抓"清尾工作流

> 场景：CDP经过调试恢复稳定后，用户要求"把没抓到的JD，应抓尽抓"
> 前置条件：CDP已确认在线（curl localhost:9222/json/version 返回正常）
> 目标：系统性地清缴之前因CDP不稳定而遗留的待核实岗位，不做重复工作

## 触发器

用户说以下任意一句时：
- "CDP稳了，把该扫尾的清一下"
- "应抓尽抓"
- "把没抓到的JD抓一抓"
- "CDP已经调教好了，补之前漏的"

## 执行流程

### Step 0: 拉当前状态

1. 读取追踪表，找到 **D区（待核实）** 中全部岗位及其来源和原因
2. 列出所有 **JD列显示"—"** 的已投岗位
3. `search_files` 扫描JD库，对比追踪表确认哪些JD文件真实存在
4. 输出清尾清单：哪些是CDP可补抓的，哪些是依赖用户手动操作的

### Step 1: CDP多平台系统扫描（按成功率排序）

按平台 **CDP JD提取成功率** 从高到低逐个扫描：

| 平台 | CDP成功率 | 扫描策略 | 预计调用 |
|------|-----------|----------|----------|
| JobsDB | ✅ 高（CDP绕过CF） | 搜索关键词+提取新岗位 | 3-5次 |
| Indeed | ✅ 高（搜索页稳定） | 不用viewjob，用搜索页右侧面板 | 3-5次 |
| 猎聘 | ✅ 中（内地岗多） | 搜香港相关 | 2-3次 |
| CTgoodJobs | ⚠️ 低（SPA架构限制） | 仅搜+提取列表，不点详情 | 2-3次 |
| 劳工处 | ✅ 高（无需登录） | 中文关键词+管理/行政类别 | 2-3次 |
| LinkedIn | ⚠️ 不可靠（见pitfall #98） | 先curl meta→有希望才CDP | 1次/岗 |

### Step 2: 处理遗留待核实岗位

逐个处理D区的"待核实"岗位：

1. **CDP可抓** → 用对应平台的最佳方法提取JD
2. **CDP失败** → 快速评估是否值得用户手动打开
   - 发布>1个月+100+申请 → 大概率已过期，直接标记❌
   - 行业壁垒明显（保险/银行/加密/奢侈品） → 直接标记❌
   - 其他 → 标记"需用户手动打开JD"，列在汇总报告
3. **已投但缺JD的** → 如果无LinkedIn ID或URL，放弃（无法追溯）
4. **新发现的岗位** → 对比追踪表查重 → 新岗位写入D区

### Step 3: 快速评估与分类

每个获取到的JD做快速评估（不需要完整9维度）：
- 行业壁垒 → ❌ 不投
- 层级不符（Assistant/Executive等） → ❌ 不投
- 方向对口+行业中立 → 写入P2候选池（按匹配度排序）
- 边界case → 分开列出供用户决策

### Step 4: 汇总报告

报告格式：
```
===== CDP清尾报告 =====

遗留待核实: X个 → 已处理Y个，跳过Z个
  新抓JD: A个（含公司/岗位/初步评估）
  已过期/壁垒: B个（直接标记❌）
  需手动: C个（列清单）

新岗位发现: D个（含平台来源）

已投缺JD: E个 → 无可追溯URL，放弃

追踪表状态: 已同步更新

待决策: ...
```

## Pitfall

- **LinkedIn JD提取不可靠**：createTarget和Page.navigate都可能失败，不要花太多调用
- **不要多次重试**：一个链接失败1次就跳过，换下一个
- **CDP断开中断**：扫到一半CDP挂了 → 当前产出已保存，汇报已有结果
- **区分"本轮新抓"和"存量复用"**：汇总报告要诚实地标注每个JD的来源
- **JobsDB详情页可能被CF拦截**：用搜索页摘要信息做评估，详情页让用户手动打开

## 调用估算

- 典型遗留岗位：2-5个（大部分已在历史session中处理完毕）
- 新岗位扫描：每个平台3-5次调用，共约20-30次
- 总时间：5-10分钟（CDP稳定前提下）
