# 批量投递工作流 (Batch Application Workflow)

> 2026-07-01 从batch3投递实战提炼

## 概念

用户将投递分为编号批次（batch1, batch2, batch3...），每批约10-15个岗位。批次编号对应时间顺序，不是优先级。

## 批次定义规则

| 优先级 | 来源 | 说明 |
|--------|------|------|
| P1全部 | 核心7/9+ | 每个batch必须包含所有P1 |
| P2过滤 | 核心6/9以上 | P2中核心三维≥6/9的进入batch |
| 暂缓 | 核心5/9 | 留在P2观望，不进batch |

典型batch大小：3(P1) + 8-10(P2过滤) = 11-13个岗位

## 追踪表格式

在"## 一、投递记录"和"## 二、岗位池"之间插入batch清单章节：

```markdown
## 1.5 Batch{N} 投递清单 (MM-DD)

> 已有简历，准备投递。P1全投 + P2核心6/9以上。

| # | 优先级 | 公司 | 岗位 | 方向 | 匹配度 | 渠道 | 简历 | 状态 |
|---|--------|------|------|------|--------|------|------|------|
| 1 | P1 | ... | ... | ... | 7/9 | LinkedIn | ✅内推版 | 待投 |
| ... |

> 暂缓: {列出核心5/9的P2岗位}
```

## 流水线

1. **检查P1/P2简历状态** — 所有进batch的岗位必须已有定制简历
2. **生成缺失简历** — 用batch-resume-generation.md批量生成
3. **在追踪表中插入batch清单** — section编号1.5（在投递记录和岗位池之间）
4. **投递** — LinkedIn岗位用Easy Apply，官网岗位告知用户手动
5. **更新状态** — 投递后将"待投"改为"已投"，从岗位池移除

## 触发词

用户说"继续batch N"、"batch N走完了"、"下一个batch"时，执行此工作流。

## 注意

- batch编号是用户约定的进度标记，不等于投递优先级
- 每个batch之间可能间隔数天（取决于简历准备和投递速度）
- batch完成后在追踪表中记录投递日期和状态
