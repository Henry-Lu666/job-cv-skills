# 批次管理与渠道扫描记录

> 2026-07-01 从实战中提炼
> 2026-07-02 更新：增加文件存在性验证

## "Batch"的正确含义

在求职渠道全景图(`08-找工作/香港求职渠道全景图.md`)的语境下，batch指**渠道扫描批次**：

| Batch | 渠道 | 状态 |
|-------|------|------|
| Batch 1 | CTgoodjobs + 劳工处 | ✅ 06-29完成 |
| Batch 2 | 中资企业8家 + LinkedIn通知 | ✅ 06-29完成 |
| Batch 2续 | 招商局 cmhk.zhiye.com + BOCHK careers.pageuppeople.com/802 | ⏳ 待执行 |
| Batch 3 | 下一批渠道（从全景图P2区选取） | ⏳ 待规划 |

**不要把batch理解为P1/P2岗位的投递批次。**

## 追踪表中的Batch记录格式

在投递追踪.md的 `## 1.5` 区域记录当前batch状态：

```markdown
### Batch3 渠道扫描待办
| # | 渠道 | URL | 状态 |
|---|------|-----|------|
| 1 | 招商局 | cmhk.zhiye.com | 待扫描 |
```

## Session恢复检查清单

每次"继续求职"开始时：

### 第一步：搜索上一个session

`session_search(sort=newest, limit=5)` 找最近的求职session，读取session末尾的"待办"或"产出"部分。

### 第二步：对比追踪表

对比session产出和追踪表，确认所有发现已记录。补录缺失项后再继续新任务。

### 第三步：验证文件存在性（2026-07-02新增）

前置session可能声称"已抓JD"或"简历已出"但实际文件未持久化。追踪表的标记可能不反映真实文件状态。**必须逐条验证：**

1. **JD文件验证**：对追踪表中标记有JD链接的岗位（P1/P2/待核实），用 `search_files(pattern='*{ID}*', target='files', path='JD库/')` 确认文件是否存在
2. **简历文件验证**：对追踪表标记"简历✅"的岗位，检查 `简历目录/` 中对应的 `.docx` 文件是否存在
3. **发现缺失时的处理**：
   - 如果CDP Chrome中对应tab还开着 → 优先通过CDP WebSocket重新提取（见pitfall #14）
   - 如果tab已关闭 → 标记为"⚠️JD文件缺失，需重抓"放入待核实区
   - 该岗位从P1/P2降级——用户会点链接发现404，不能让断链留在表格中
4. **报告诚实性**：在Stage 5汇总报告中标注每个产出来源——"本轮新抓" vs "存量文件" vs "推理/假设"

## 常见错误

- ❌ 把batch理解为投递批次 → 用户纠正
- ❌ 新session开始时不检查上一个session的产出 → 基信安康/皇固漏记录
- ❌ 在当前对话中找batch定义 → 应该用session_search查历史session
- ❌ 相信追踪表的"✅"标记而不验证文件存在 → 港沪联盟控股"已抓JD"但无文件
