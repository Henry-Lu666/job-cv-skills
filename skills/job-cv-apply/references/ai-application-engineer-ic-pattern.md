# AI应用开发工程师IC角色 — 评估与定制模式

> 2026-07-03 验证：奇富科技QFIN "AI应用开发工程师（大模型方向-香港）"
> 这是一类不同于三个主力方向（BD/AI落地推行/数字化转型）的新模式

## 角色特征

| 维度 | 典型描述 |
|------|---------|
| **Title** | AI应用开发工程师 / LLM Engineer / AI Engineer |
| **级别** | IC（Individual Contributor），非管理岗 |
| **经验要求** | 应届~3年，MSc学历多为硬门槛 |
| **核心技能** | RAG、Agent编排、Prompt Engineering、AI Native开发（Claude Code/Codex） |
| **面试重点** | GitHub作品、AI应用上线经验、工程化思维 |
| **典型薪资** | HKD 25-50K·15薪 |

## 与三个主力方向的关键差异

| 对比项 | BD/AI落地/数字化 | AI应用开发工程师IC |
|--------|-----------------|-------------------|
| 角色定位 | 管理/项目推动 | 个人技术贡献者 |
| 经验年限 | 12年管理经验是优势 | 12年管理经验需要重新包装 |
| 面试侧重 | 案例/Star故事/管理经验 | 代码/项目/GitHub作品 |
| 简历侧重 | 团队管理、SOP、扭亏 | 技术栈、项目架构、工程实践 |
| 方向标签 | ①BD ②数字化 ③运营 | AI工程 |

## 匹配评估要点

**该模式对用户的评估校准**（相对于三个主力方向）：

| 维度 | 评分逻辑 | 典型校准 |
|------|---------|---------|
| 技能匹配 | 实际权重最高 | 应有真实RAG/Agent/Prompt项目支撑 |
| 行业匹配 | **权重降低** — AI工程化跨行业迁移性强 | 金融科技vs汽车制造不再是大问题 |
| 职能匹配 | 看IC vs 管理的选择意愿 | 面试需解释转型动机 |
| 经验年限 | 只看AI相关年限 | 12年管理经验不扣分也不加分，视为"业务理解"加成 |

## 简历定制模式

### 求职目标方向
`AI 应用开发 / 大模型工程化 / Agent 系统 · 香港`

### 自我评价定位
**卖点公式：** `MSc Applied AI + 具体项目经验(RAG/Agent/MCP) + 业务理解(12年管理=懂用户需求)`

- 突出技术能力栈：RAG流水线、Agent编排、MCP协议、Prompt Engineering
- 突出工具使用：Claude Code、Codex、GPT——JD往往明确要求
- **12年管理经验重新包装为"理解真实业务需求的差异化优势"**
- 强调GitHub作品可展示

### 关键调整
1. ❌ 不强调团队规模（650人）、管理年限
2. ✅ 强调个人项目贡献、技术选型决策、代码质量
3. ✅ 展示AI Native开发流程认知
4. ⚠️ 后端工程经验（FastAPI/Docker/异步）不足时坦诚但强调学习能力

## 面试风险项

| 风险 | 应对策略 |
|------|---------|
| "管理转IC，能适应吗？" | 强调AI方向是主动选择，12年业务经验让我的AI方案更落地 |
| "后端工程经验不足" | 展示RAG/Agent项目中的工程部分，强调AI Native开发能力 |
| "作品集不够工程化" | 强调审计RAG系统的端到端架构（ChromaDB+Flask+LLM部署） |
| "行业差异" | AI应用开发技能跨行业可迁移，金融科技vs制造不是障碍 |
