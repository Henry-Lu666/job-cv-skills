# 求职全流程管线

> 本skill定义Stage 0-5的完整求职管线框架，含场景判断、操作流程和关键教训。

---

## 管线场景判断

执行"走一遍"前，先判断当前场景：

| 场景 | 典型触发条件 | Stage 2-3 策略 |
|------|-------------|---------------|
| **🆕 首次扫描** | 新启动求职，追踪表为空或少量岗位 | 全量JD提取→评估→P1出简历 |
| **🔁 日常巡检**（最常见） | 追踪表已有完整P1/P2评分 | 快速通道：Stage 2降为存活验证，Stage 3跳过 |
| **📬 发现新强推** | Gmail/CDP扫到1-3个新岗位 | 仅对新岗位走L2-L3核实 |

## 输出管线（Stage 0-5）

```
Stage 0: 时间校准
└── terminal("date") 确认当前时间

Stage 1: 扫描（L1）
├── Gmail扫描（MCP）
├── CDP多平台扫描（浏览器串行扫各平台）
├── 结果标注"⚠️未核实"
├── 查重追踪表
└── 输出：待核实岗位清单

Stage 1.5: 签证友好度筛查
├── 检查JD中的签证/工作许可要求
├── 不友好信号："right to work""visa sponsorship not available"
├── IANG友好信号："visa sponsorship available""open to all nationalities"
└── 输出：签证友好度标签

Stage 2: 核实（L2→L3）
├── 批量抓取JD（按平台级联策略）
├── JD保存到JD库（含中文翻译）
├── jd-inspector 执行：坑点扫描→门槛审查→9维度评分
├── 主agent验证JD文件完整性+追踪表更新
├── 核心三维<5/9 或 门槛不通过→归入观望/划掉
└── 输出：已核实岗位清单（P1/P2/观望/划掉）

Stage 3: 简历定制+投递
├── 验证岗位存活（检查不是404/已关闭）
├── 读取base简历，生成定制版
├── 可选：生成Cover Letter
├── 投递前需用户确认（不可跳过）
└── 输出：定制版简历.docx + 投递记录

Stage 4: 投后追踪
├── 更新追踪表投递状态
├── 设置提醒（7天无反馈follow-up）
└── 输出：投递状态更新

Stage 5: 汇总报告
├── 所有操作提醒同步写入追踪表
├── 输出面向用户决策的报告
└── 输出：用户可操作的摘要
```

## 平台限制清单

| 平台 | JD提取方法 | 限制 |
|------|-----------|------|
| LinkedIn | CDP WebSocket导航已登录tab | 反爬严格，新tab被block |
| JobsDB | 推荐邮件摘要+L2 | CF拦截详情页 |
| Indeed | 搜索页右侧面板 | viewjob被CF拦截 |
| CTgoodJobs | browser_navigate | 结构简单 |
| Glassdoor | browser_navigate | 推荐邮件为主 |
| 猎聘 | browser_navigate | 中文平台 |
| 劳工处 | browser_navigate | 简易页面 |
| 企业ATS（SuccessFactors等） | browser_navigate直接提取 | 无特殊反爬 |

## 关键教训（Pitfalls精选）

### 岗位评估类

1. **L1结果≠已核实匹配** — Gmail扫描的"强推"、平台标题评分、百分比——全部是关键词粗筛。必须走L2/L3核实后才能下结论。
2. **岗位匹配度不能只看方向标签** — "方向对口≠能力匹配"。评估时必须逐条检查硬性要求（行业深度、语言、年限、技术栈）。
3. **行业壁垒是刻度不是开关** — 区分核心业务岗（需资质）vs 后台赋能岗（方法论可迁移）。软壁垒标"⚠️面试风险"不直接毙。
4. **"AI in title, non-AI core"陷阱** — 含"(AI)"的岗位实际可能是金融风控等其他职能。必须先看Service Line/Department确认。
5. **P1候选必须重新核实JD和评估** — 追踪表历史评分可能基于不完整信息。P1前必须重新抓取完整JD做L3核实。
6. **银行/金融Director级10年+经验是硬门槛** — 即使title含甜蜜区方向，行业经验年限无法逾越。

### 技能执行类

7. **已登录tab优先** — CDP模式下先检查已有tab，用Runtime.evaluate直接操作登录tab。
8. **简历定制前必须验证岗位存活** — 打开JD页确认不是404/已关闭才进入定制。
9. **中英双语bullet必须同步生成** — 格式一致性检查（无主语短句，keyword: content格式）。
10. **python-docx修改后必须save+验证** — 不要依赖内存中的修改做后续操作。
11. **简历内容幻觉红线** — 任何百分比/评级/排名数据必须确认有实测支撑。

### 流程管理类

12. **追踪表和JD库必须同步** — 有评估记录的岗位必须进追踪表，不留孤儿记录。
13. **"走一遍"模式投递必须等用户确认** — 投递权在用户，agent不可擅自操作。
14. **追踪表不记录JD要求，JD库才是JD的归属** — 追踪表功能是统计投递状态。
15. **CDP模式下LinkedIn JD提取** — browser_navigate会触发反爬→用CDP WebSocket导航现有登录tab。
16. **同一公司不同岗位的粤语要求可能不同** — 必须逐岗位读JD确认，不能因A岗有粤语要求推断B岗也有。
17. **同公司投多个岗位时，简历bullet必须差异化** — 同一HR看到一模一样的bullet=模板海投。
18. **子Agent输出不是事实，必须验证来源** — 主Agent必须至少做一次快速来源验证。
19. **ATS排序规则因系统而异** — 确认公司用什么ATS，再查其默认排序规则。
20. **输出型daily cron用弹性间隔** — `every 24h`替代固定时间，避免agent断线错过执行。

## JD提取级联策略

```
提取JD优先级：
1. CDP WebSocket → 用户已登录tab（LinkedIn/JobsDB）
2. browser_navigate → 新tab（可获取cookie的站点）
3. 搜索页右侧面板（Indeed/部分JobsDB）
4. 推荐邮件正文提取（最低优先级）
```

## 同公司多岗位投递间隔建议

| 场景 | 建议间隔 | 说明 |
|------|---------|------|
| 同一公司**不同部门** | ≥3天 | 不同HR团队处理 |
| 同一公司**同一部门** | ≥7天 | 同一HR看到关联申请 |
| 必须在短间隔内投 | — | 在Cover Letter中说明为什么两个岗都感兴趣 |
